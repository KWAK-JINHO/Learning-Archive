# Index Tuning

인덱스 튜닝의 핵심 목표는 **테이블 액세스를 최소화하는 것**이다.

대부분의 SQL 성능 문제는 다음 두 가지에서 발생한다.

- 인덱스 스캔량이 많음
- 인덱스 이후 테이블 액세스가 많음

따라서 인덱스 튜닝은 해당 두 가지를 줄이는 방향으로 진행한다.

## 테이블 액세스 최소화

인덱스를 사용하더라도 결국 테이블을 읽어야 한다면 랜덤 I/O가 발생하여 성능이 저하될 수 있다.  
따라서 가능한 한 인덱스만으로 조회한다.

## 인덱스 손익분기점

인덱스는 항상 빠른 것이 아니다.  
데이터 조회 비율이 높아지면 오히려 **Table Full Scan이 더 빠를 수 있다.**
즉, 인덱스는 선택도가 높을 때 유리하다.

## 부분범위 처리

![OLTP](./oltp_olap.md#oltp) 시스템에서는 **부분범위 처리** 가 중요하다.

예시

```sql
SELECT *
FROM orders
ORDER BY order_date LIMIT 10;
```

DB는 전체 데이터를 읽지 않고 필요한 만큼만 읽는다. OLTP 환경에서는 이런 패턴이 중요하다.

## 인덱스 스캔 효율

인덱스 스캔 성능은 조건절 구성 방식에 크게 영향을 받는다.

### 1. 등치 조건 (=)이 가장 빠르다.

예시) `WHERE user_id = 10`  
인덱스 B+Tree에서 한 번의 탐색으로 바로 접근 가능하다.

### 2. Range 조건은 스캔 범위가 넓어진다.

예시)`WHERE age BETWEEN 20 AND 30`  
20 위치 찾기 -> 30까지 스캔 (넓어질수로 성능 떨어진다.)

### 3. 복합 인텍스 컬럼 순서 와 구성

중간 컬럼이 빠지면 사용할 수 없고 `INDEX (A, B, C)` 와 같은 인덱스에서 `WHERE B = ?` 같은 조건은 정상적으로 인덱스를 사용하기 어렵다.

### 4. 조건절 구성

SQL 조건절은 1. 인덱스를 탐색하는 조건 2. 탐색 후 결과 걸러내는 조건 두가지로 나뉜다.

예시) `WHERE user_id = 10 AND age > 20`  
이 때 인덱스를 탐색하는 조건이 많을수록 성능이 좋다.

### 5. BETWEEN vs IN-LIST

Range 조건을 여러 개의 등치 조건으로 분해하면 성능이 좋아질 수 있다.

예시

`WHERE id BETWEEN 1 AND 3` -> `WHERE id IN (1,2,3)`
Range Scan 대신 여러 번의 Index 탐색으로 처리 가능.

### 6. 커버링 인덱스

쿼리에 필요한 모든 컬럼이 인덱스에 포함되어 있어  
**테이블 접근 없이 인덱스만으로 결과를 반환할 수 있는 경우**를 말한다.

예시

```sql
SELECT name, age
FROM users
WHERE name = 'kim';
```

인덱스 (name, age) 경우 DB는 테이블을 접근하지 않고 인덱스만 사용하여 결과를 반환

## 인덱스 설계 시 고려 사항

인덱스는 많을수록 좋은 것이 아니다.

### 인덱스가 많아질 때 발생하는 문제

- INSERT 성능 저하
- UPDATE 성능 저하
- DELETE 성능 저하
- 저장 공간 증가

### 인덱시 설계 기본 원칙

1. 자주 사용하는 조건 컬럼
2. = 조건 컬럼을 앞쪽에 배치
3. 선택도가 높은 컬럼 우선
4. 정렬 컬럼 고려

---

## 인덱스 리빌드

대량의 INSERT / UPDATE / DELETE가 발생하면  
인덱스 내부에 **Fragmentation(단편화)** 이 발생할 수 있다.

이 경우 인덱스를 재구성(Rebuild)하면 디스크 공간과 탐색 성능을 개선할 수 있다.

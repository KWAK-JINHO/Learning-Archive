# MySQL

## Architecture

InnoDB는 **Clustered Index 기반 저장 구조**를 사용하는 스토리지 엔진이다.   
테이블 데이터 자체가 **Primary Key 기준으로 정렬된 B+Tree** 구조로 저장된다.  
즉, Primary Key Index = Table Data 이다.

### 특징

- 테이블 데이터가 Primary Key 기준으로 물리적으로 정렬된다.
- PK 조회 성능이 매우 빠르다.
- Secondary Index는 실제 데이터 대신 **Primary Key 값을 참조**한다.

---

## InnoDB의 탐색 구조

1. Secondary Index Scan  
   보조 인덱스(예: 이름, 이메일 등)에서 조건에 맞는 값을 찾습니다. 이 인덱스의 리프 노드에는 실제 데이터의 주소가 아니라 PK 값이 들어있다.

2. PK Index Lookup  
   위에서 찾아낸 PK 값을 가지고 다시 **PK 인덱스(B-Tree)**를 루트부터 리프까지 타고 내려갑니다.

3. Data Access  
   PK 인덱스의 리프 노드에 도달하면 실제 행을 읽어온다.

InnoDB가 이와같이 두번이나 Look Up을 하는 이유는

- **데이터 이동 시 유리**  
  데이터가 업데이트되어 물리적 위치가 바뀌어도(Page Split 등), 보조 인덱스들이 들고 있는 PK 값은 변하지 않는다.   
  즉, 모든 보조 인덱스를 일일이 수정할 필요가 없다.
- **인덱스 크기**  
  PK가 아주 크지 않다면 보조 인덱스의 크기를 일정하게 유지할 수 있다.

---

## Covering Index

쿼리에 필요한 모든 컬럼이 인덱스에 포함된 경우 테이블 접근 없이 인덱스만으로 쿼리를 처리할 수 있다.  
이를 **Covering Index**라고 하며 Random I/O가 감소하는 특징이 있다.

---

## GROUP BY 최적화

MySQL에서는 GROUP BY 수행 시 인덱스를 사용하지 못하면 filesort가 발생한다.

- filesort: 메모리나 디스크의 sort buffer에서 직접 데이터를 정렬하는 과정. 메모리가 부족하면 디스크에서 정렬한다.

---

## ORDER BY NULL

GROUP BY 수행 시 정렬이 필요 없는 경우 다음과 같이 작성하여 불필요한 정렬을 제거할 수 있다.

```sql
SELECT type, COUNT(*)
FROM table
GROUP BY type
ORDER BY NULL;
```

- 불필요한 filesort 제거
- GROUP BY 성능 개선

---

## Clustered Index와 페이징

InnoDB는 PK 기준으로 데이터가 정렬되어 있기 때문에 다음과 같은 쿼리는 효율적으로 수행된다.

```sql
SELECT *
FROM user
WHERE id > 100
ORDER BY id LIMIT 10;
```

이 방식은 'No Offset Pagination' 라고 한다.

- OFFSET Scan을 제거
- 대용량 데이터에서 성능 개선 의 장점을 가진다.

## MySQL vs PostgreSQL

| 항목              | MySQL (InnoDB)  | PostgreSQL     |
|-----------------|-----------------|----------------|
| 테이블 저장 구조       | Clustered Index | Heap Table     |
| 데이터 정렬 기준       | Primary Key     | 없음             |
| Secondary Index | PK 참조           | TID 참조         |
| PK 조회 성능        | 매우 빠름           | Heap 접근 필요     |
| MVCC 방식         | Undo Log 기반     | Row Version 기반 |
| UPDATE 처리       | 기존 row 수정       | 새 tuple 생성     |
| Vacuum 필요       | 없음              | 필요             |

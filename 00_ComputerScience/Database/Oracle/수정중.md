# Oracle

## 클러스터링 팩터 (Clustering Factor, CF)

클러스터링 팩터는 **인덱스에 저장된 순서와 실제 테이블 데이터가 저장된 물리적 순서가 얼마나 비슷한지**를 나타내는 지표이다.

- CF가 낮을수록 인덱스를 따라 테이블을 읽을 때 블록 I/O가 적게 발생한다.
- CF가 높을수록 같은 건수를 조회해도 더 많은 블록을 읽게 되어 비효율적이다.
- 인덱스 손익분기점에 큰 영향을 준다.

즉, 인덱스 스캔 후 테이블 액세스가 많이 발생하는 환경에서는  
**클러스터링 팩터가 성능에 중요한 영향을 준다.**

---

## IOT (Index Organized Table)

IOT는 **테이블 데이터를 별도 힙 영역에 저장하지 않고, 인덱스 구조 자체에 함께 저장하는 방식**이다.

특징

- 테이블이 인덱스 구조로 저장된다.
- PK 기반 조회 성능이 매우 뛰어나다.
- 일반적인 테이블처럼 테이블 액세스가 추가로 발생하지 않는다.
- 랜덤 액세스를 줄일 수 있다.

주의점

- 일반적인 힙 테이블과 특성이 다르다.
- 특정 조회 패턴에는 매우 유리하지만 범용적으로 쓰기에는 제약이 있다.
- 극단적인 물리 설계 방식에 가깝다.

---

# 2. Oracle 메모리 구조

## SGA (System Global Area)

SGA는 Oracle 인스턴스가 사용하는 **공유 메모리 영역**이다.

대표 구성 요소

- Database Buffer Cache
- Shared Pool
- Redo Log Buffer

특징

- 여러 프로세스가 함께 사용하는 공유 메모리 영역
- SQL 실행, 캐시, 로그 기록 등에 사용된다.

---

## PGA (Program Global Area)

PGA는 Oracle 서버 프로세스가 사용하는 **개별 메모리 영역**이다.

특징

- 프로세스별로 독립적으로 사용된다.
- 정렬(Sort), 해시(Hash), 세션별 작업 영역 등에 사용된다.
- 공유되지 않는다.

튜닝 관점

- Sort / Hash Join / Group By 같은 작업은 PGA를 사용한다.
- PGA 공간이 부족하면 디스크의 **TEMP Tablespace** 를 사용하게 된다.
- 이 경우 성능이 크게 저하될 수 있다.

즉, Oracle에서 소트/해시 작업이 느려지는 주요 원인 중 하나는  
**PGA 부족으로 인한 TEMP 사용**이다.

---

## TEMP Tablespace

TEMP Tablespace는 정렬, 해시 조인 등 작업 도중  
메모리(PGA)만으로 처리할 수 없을 때 사용하는 임시 공간이다.

사용되는 대표 작업

- Sort
- Hash Join
- Group By
- Order By

특징

- 디스크 기반 작업이 발생한다.
- TEMP 사용량이 많아질수록 성능 저하 가능성이 크다.

---

# 3. Redo / Undo

## Redo Log

Redo Log는 Oracle에서 발생한 변경 사항을 기록하는 로그이다.

용도

- Database Recovery
- Cache Recovery
- Fast Commit 지원

특징

- DML 수행 시 Redo 로그가 생성된다.
- 장애 발생 시 변경 내역을 재적용하는 데 사용된다.
- 성능에도 영향을 준다.

---

## Undo

Undo는 변경 이전 값을 저장하는 정보이다.

용도

1. Transaction Rollback
2. Transaction Recovery
3. Read Consistency 보장

특징

- DML 수행 시 Undo 정보가 함께 생성된다.
- 쿼리 수행 시 일관된 읽기를 보장하는 데 사용된다.

---

## Read Consistency

Oracle은 쿼리 수행 중 다른 트랜잭션이 데이터를 변경하더라도  
쿼리 시작 시점 기준의 일관된 데이터를 읽을 수 있도록 보장한다.

이때 Undo 데이터를 활용한다.

즉,

- 현재 블록이 변경되었더라도
- 쿼리 시작 시점 이후 변경이라면
- Undo를 이용해 이전 버전을 재구성하여 읽는다.

---

## MVCC (Multi-Version Concurrency Control)

Oracle은 다중 버전 읽기 일관성을 제공한다.

### Current Mode

현재 시점의 최신 블록을 읽는 방식

### Consistent Mode

쿼리 시작 시점 기준으로 일관된 데이터를 읽기 위해  
Undo를 반영한 복사본을 읽는 방식

특징

- SELECT는 일관된 읽기를 위해 Consistent Mode를 사용한다.
- DML은 대상을 찾을 때 Consistent Read를 활용하고, 실제 변경은 Current 블록에 적용한다.

---

# 4. Oracle 백그라운드 프로세스

## LGWR (Log Writer)

Redo Log Buffer의 내용을 Redo Log File에 기록하는 프로세스이다.

특징

- COMMIT 시점과 밀접하게 관련된다.
- 변경 내용을 로그 파일에 기록하여 영속성을 보장한다.

---

## DBWR (Database Writer)

Buffer Cache의 변경된 블록(Dirty Buffer)을 Datafile에 기록하는 프로세스이다.

특징

- 버퍼 캐시의 변경 내용을 데이터 파일에 반영한다.
- 커밋이 곧바로 데이터 파일 기록을 의미하지는 않는다.

---

# 5. 커밋의 내부 메커니즘

Oracle에서 커밋이 발생할 때의 핵심 흐름은 다음과 같다.

1. DML 실행
2. Redo Log Buffer에 변경 사항 기록
3. Buffer Cache에서 블록 변경
4. COMMIT 수행
5. LGWR가 Redo Log Buffer 내용을 로그 파일에 기록
6. 이후 DBWR가 변경된 블록을 Datafile에 반영

핵심 포인트

- COMMIT은 주로 Redo 기록 완료와 관련 있다.
- 데이터 파일 반영은 나중에 일괄 수행될 수 있다.
- 그래서 Oracle은 빠른 커밋이 가능하다.

---

# 6. Direct Path Insert

## Direct Path Insert란

일반 INSERT와 달리 버퍼 캐시를 적극적으로 거치지 않고  
데이터 파일에 직접 기록하는 방식이다.

특징

1. Freelist를 참조하지 않고 HWM 바깥 영역에 순차적으로 입력
2. 버퍼 캐시를 탐색하지 않음
3. 버퍼 캐시에 적재하지 않고 데이터 파일에 직접 기록
4. Undo 로깅을 최소화하거나 하지 않음
5. Redo 로깅 선택 가능

장점

- 대량 INSERT에서 매우 빠르다.

주의사항

1. TM Lock이 걸릴 수 있다.
2. 커밋 전까지 다른 트랜잭션의 DML이 제한될 수 있다.
3. 테이블의 빈 공간을 재사용하지 못할 수 있다.

---

# 7. Lock

## Oracle Lock 개요

Oracle은 공유 리소스와 사용자 데이터를 보호하기 위해  
여러 종류의 Lock을 사용한다.

대표 예시

- DML Lock
- DDL Lock
- Latch
- Buffer Lock

---

## DML Row Lock

특징

- 두 트랜잭션이 같은 로우를 동시에 변경하지 못하도록 방지한다.
- UPDATE / DELETE 시 배타적으로 동작한다.
- INSERT 시에는 Unique 인덱스 충돌 상황에서 경합이 발생할 수 있다.

튜닝 포인트

- 온라인 트랜잭션 환경에서는 Lock을 오래 유지하지 않도록 해야 한다.
- 커밋 시점이 지나치게 늦으면 경합이 심해질 수 있다.

---

## DML Table Lock

Oracle은 Row Lock을 설정하기 전에  
테이블 수준의 Lock도 함께 사용한다.

주의할 점

- Table Lock은 테이블 전체를 막는다는 의미라기보다 현재 이 테이블에서 어떤 작업이 수행 중인지 나타내는 표시 성격이 강하다.

---

# 8. Oracle 힌트

## LEADING 힌트

여러 테이블 조인 시 조인 순서를 제어하기 위한 힌트이다.

특징

- 세 개 이상의 테이블 조인 시 조인 순서를 명시할 수 있다.
- 기존 SQL 구조를 크게 바꾸지 않고 실행계획을 유도할 때 사용한다.

단, 힌트는 최후의 수단에 가깝고  
통계정보, 인덱스, SQL 구조 개선이 우선이다.

---

# 9. 서브쿼리 Unnesting

Unnesting은 서브쿼리를 별도로 수행하지 않고  
조인 형태로 변환하여 실행하는 최적화 기법이다.

배경

- 메인 쿼리의 각 행마다 서브쿼리가 반복 실행되면 비효율적일 수 있다.
- 이를 조인으로 풀어서 더 효율적인 실행계획을 만들 수 있다.

---

## Semi Join

Semi Join은 첫 번째 테이블의 행만 반환하면서  
두 번째 테이블은 **조건 만족 여부만 확인**하는 조인 방식이다.

주로 다음 구문과 관련된다.

- `IN`
- `EXISTS`

장점

- 불필요한 데이터 반환을 줄일 수 있다.
- 조인 결과 집합을 단순화할 수 있다.

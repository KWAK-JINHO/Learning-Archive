# 객체 (Object)

## 객체란

객체는 상태(State)와 행동(Behavior)을 가지며, 스스로 책임을 수행하는 단위이다.

- 단순 데이터 묶음이 아니라 행동을 포함한 실행 단위
- 객체지향에서 핵심은 "데이터"가 아니라 "행동"

---

## 객체의 구성 요소

### 1) 상태 (State)

객체가 가지는 값이며 시간에 따라 변경된다.

예시

```markdown
class Account { int balance; }
```

### 2) 행동

상태를 변경하거나 조회하는 기능이다. 객체는 오직 행동을 통해서만 상태를 변경해야 한다.

예시

```markdown
void deposit(int amount) { balance += amount; }
```

### 3) 식별자

객체를 구별하는 고유성이다. 값이 같더라도 서로 다른 객체일 수 있다.

예시

`new Account() != new Account()`

---

## 객체 설계의 핵심 원칙

### 1) 상태보다 행동이 먼저다

협력에 필요한 행동을 먼저 정의하고, 그에 따라 상태를 결정해야 한다.

- 상태 먼저 설계하면 캡슐화가 깨지고 유지보수가 어려워진다
- 행동 중심 설계가 객체지향의 핵심

### 2) 객체는 자율적이어야 한다

객체는 자신의 상태를 스스로 관리해야 하며 외부에서 직접 변경하면 안된다.

나쁜 예

`account.balance += 1000;`

좋은 예

`account.deposit(1000);`

---

### 3) 객체는 협력한다

객체는 단독으로 존재하지 않고 메시지를 통해 서로 상호작용한다.

예시

`order.pay(payment);`

---

## 객체는 블랙박스다

- 내부 구현은 숨기고
- 외부에는 인터페이스만 제공한다

핵심:
객체는 "어떻게"가 아니라 "무엇을 하는지"만 외부에 드러낸다

---

## Query vs Command

### Query (조회)

상태를 변경하지 않고 결과만 반환한다.

예시

`int getBalance()`

### Command (명령)

상태를 변경한다.

예시

`deposit(1000)`

---

## 객체 설계 시 흔한 실수

### 1) 상태 중심 설계

```markdown
class User { String name; int age; }
```

문제:

- 단순 데이터 덩어리가 됨
- 로직이 외부로 분리됨

### 2) Getter 남발

- 캡슐화가 깨짐
- 객체의 의미가 사라짐

### 3) 객체를 DB처럼 생각

객체는 테이블이 아니라 "행동하는 주체"이다

---

## 요약

객체는 상태가 아니라 행동과 책임으로 정의된다

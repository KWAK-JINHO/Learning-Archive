# 객체지향 설계 프로세스

## 설계 프로세스란 무엇인가

객체지향 설계 프로세스는 요구사항을 객체, 책임, 협력 구조로 변환하는 과정이다.

핵심

- 기능 목록을 바로 클래스 설계로 옮기지 않는다
- 먼저 문제 영역을 이해하고, 객체들이 어떻게 협력할지부터 설계한다

---

## 기능 중심 설계 vs 구조 중심 설계

### 기능 중심 설계

사용자에게 어떤 기능을 제공할지에 초점을 맞춘다.

예

- 회원가입
- 결제
- 주문조회

문제점

- 요구사항이 바뀔 때 구조가 쉽게 흔들린다
- 기능별로 코드를 덧붙이다 보면 복잡도가 빠르게 증가한다

### 구조 중심 설계

시스템이 어떤 객체와 관계로 이루어지는지에 초점을 맞춘다.

예

- 주문은 결제를 가진다
- 회원은 주문을 생성한다
- 알림은 다양한 채널로 전송된다

장점

- 변경에 강하다
- 기능이 추가되어도 기존 구조를 재사용하기 쉽다

핵심 객체지향 설계는 기능보다 구조를 먼저 안정적으로 잡는 것이 중요하다

---

## 유스케이스 모델링

유스케이스는 사용자가 시스템을 통해 달성하려는 목표를 기준으로 기능을 정리하는 방법이다.

예

- 사용자는 상품을 주문한다
- 관리자는 주문 상태를 변경한다
- 사용자는 결제를 취소한다

특징

- 단순한 기능 목록보다 사용자 목표 중심이다
- UI나 내부 구현 세부사항을 바로 넣지 않는다
- 시스템이 어떤 일을 해야 하는지 큰 흐름을 파악하는 데 적합하다

핵심 유스케이스는 객체 구조를 직접 만들기 위한 도구라기보다, 시스템의 요구를 이해하기 위한 출발점이다

---

## 도메인 모델링

도메인 모델링은 문제 영역의 개념, 관계, 규칙을 구조화하는 과정이다.

예

- 회원
- 주문
- 결제
- 상품

핵심 질문

- 이 도메인에는 어떤 개념이 존재하는가
- 각 개념은 어떤 관계를 가지는가
- 어떤 규칙과 제약이 있는가

목적

- 사용자와 개발자가 같은 개념을 바라보게 한다
- 기능이 바뀌어도 쉽게 흔들리지 않는 안정적인 구조를 만든다

---

## 책임 주도 설계

책임 주도 설계는 객체를 먼저 찾는 것이 아니라, 책임을 먼저 찾고 그 책임을 수행할 객체를 배치하는 방식이다.

순서

1. 시스템이 해야 할 일을 식별한다
2. 이를 더 작은 책임으로 나눈다
3. 각 책임을 가장 잘 수행할 객체를 찾는다
4. 객체 간 협력 관계를 정의한다

협력에 필요한 책임이 객체를 만들고, 객체의 책임이 상태를 결정한다

---

## 설계의 기본 흐름

객체지향 설계는 보통 다음 흐름으로 진행된다.

1. 요구사항을 유스케이스로 정리한다
2. 도메인 개념을 식별한다
3. 객체 간 협력을 설계한다
4. 각 객체의 책임을 나눈다
5. 공용 인터페이스를 정의한다
6. 필요한 구현과 상태를 내부에 배치한다

핵심 협력 → 책임 → 객체 → 상태

---

## 개념 관점 / 명세 관점 / 구현 관점

### 개념 관점

문제 영역의 개념과 관계를 본다.

예

- 주문
- 결제
- 회원

### 명세 관점

객체가 외부에 무엇을 제공하는지 본다.

예

- 주문은 결제를 요청할 수 있다
- 결제는 승인할 수 있다

### 구현 관점

실제로 동작하는 코드와 내부 로직을 본다.

예

- 필드
- 메서드 구현
- 예외 처리

핵심 좋은 설계는 이 세 관점을 구분하면서도 하나의 코드 안에 자연스럽게 녹여내야 한다

---

## 설계 시 흔한 실수

### 1) 기능 목록을 바로 클래스 목록으로 바꾸는 것

예

- 주문조회Service
- 주문생성Service
- 주문취소Service

문제

- 객체가 아니라 기능 단위로만 생각하게 된다

### 2) 상태를 먼저 설계하는 것

문제:

- 캡슐화가 깨지고 객체가 데이터 덩어리가 된다

### 3) 구현을 너무 빨리 시작하는 것

문제:

- 협력과 책임이 정리되기 전에 코드가 굳어버린다

---

## 요약

객체지향 설계 프로세스의 핵심은 요구사항을 바로 코드로 옮기지 않고, 유스케이스와 도메인 모델을 통해 구조를 이해한 뒤 협력과 책임 중심으로 객체를 설계하는 것이다

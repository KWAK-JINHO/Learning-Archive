# CV(클래스 변수) vs IV(인스턴스 변수)와 static

## CV vs IV
### Instance Variable
객체 생성 시 Heap 영역에 생성됨. 인스턴스마다 독립적인 상태를 가진다.

### Class Variable
클래스 로딩 시 메서드(Static) 영역에 생성됨. 모든 인스턴스가 공유한다. static 키워드 사용.

### static의 남용
- Global State 문제: 애플리케이션 어디서든 접근 가능하므로 상태 추적이 어렵고, 클래스 간 결합도가 높아져 유지보수가 힘들어짐.
- Thread-safe 문제: 여러 스레드가 동시에 공유 자원인 static 변수에 접근하여 수정할 경우 데이터 정합성이 깨지는 race condition 발생
- 메모리 효율: GC의 관리 대상에서 벗어나거나 우선순위가 낮아 프로그램 종료 시까지 메모리를 점유한다.
- 테스트의 어려움: 객체 지향의 특징인 다형성을 활용할 수 없고 Mock 객체 대체가 어려워 단위 테스트 작성이 까다로움.

## static vs Spring 싱글톤
[싱글톤 패턴 정리](../../03_Architecture/DesignPattern/singleton_pattern.md)
- 구현 방식: static은 언어 차원의 키워드로 정적 메모리에 고정되지만, 스프링 싱글톤은 일반 객체를 스프링 컨테이너가 관리함.
- 다형성: static 메서드나 변수는 인터페이스 구현이나 상속이 불가능하지만, 싱글톤 빈은 인터페이스 기반의 의존성 주입이 가능.
- life cycle: static은 클래스 로딩 시점에 생성되지만, 싱글톤 빈은 스프링 컨테이너에 의해 생성, 의존성 주입, 초기화, 소멸 과정이 제어된다.

로직이 없는 유틸리티 성격의 메서드(Math 등)가 아니라면, 
상태를 가진 변수는 static보다 스프링의 싱글톤 빈으로 관리하는 것이 객체 지향 설계와 멀티스레드 환경에서 안전함.
